# PPU: высокоуровневая структура проекта

## Назначение

PPU (PLC Polling Utility) - небольшой сервис для периодического чтения регистров PLC по Modbus TCP и публикации последнего результата через HTTP API.

Система сейчас решает три основные задачи:

- подключается к PLC по Modbus TCP;
- периодически читает выбранный диапазон регистров;
- хранит последний результат чтения в памяти и отдаёт его через HTTP.

## Основные компоненты

### HTTP API

Точка входа приложения находится в `Ppu/Program.cs`.

Компонент отвечает за:

- запуск ASP.NET Core Minimal API;
- регистрацию зависимостей;
- публикацию OpenAPI-описания;
- предоставление HTTP-эндпоинтов.

Основные эндпоинты:

- `GET /` - базовая информация о сервисе;
- `GET /health` - простой health-check;
- `GET /last-read` - последний результат чтения PLC из памяти;
- `GET /openapi/v1.json` - OpenAPI-схема.

### Фоновый опрос PLC

`PollingWorker` - фоновый сервис, который запускается вместе с приложением.

Он выполняет основной рабочий цикл:

- ждёт интервал из конфигурации;
- вызывает чтение PLC через `IPlcReader`;
- получает `RawReadResult`;
- сохраняет результат в `LastReadStore`;
- пишет технические события в лог.

### Чтение PLC

`IPlcReader` - интерфейс чтения данных.

`PlcReaderService` - текущая реализация чтения PLC через `FluentModbus`.

Компонент использует настройки из `PlcReaderOptions`:

- адрес PLC;
- порт;
- Unit ID;
- Function Code;
- стартовый адрес;
- количество регистров;
- интервал опроса.

Поддерживаемые функции Modbus:

- FC03 `ReadHoldingRegisters`;
- FC04 `ReadInputRegisters`.

### Хранилище последнего чтения

`LastReadStore` - простое in-memory хранилище.

Оно содержит только последний результат чтения PLC. Именно из него эндпоинт `/last-read` отдаёт данные внешним клиентам.

Важно: данные в `LastReadStore` живут только в памяти процесса. После перезапуска приложения последнее значение теряется.

### Доменная модель

Доменный слой сейчас минимальный.

Основные структуры:

- `RawReadResult` - результат одного чтения PLC;
- `ModbusFunctionCode` - поддерживаемые коды функций Modbus.

`RawReadResult` используется как единая структура результата одного обращения к PLC. Её создаёт `PlcReaderService`, затем `PollingWorker` передаёт её в `LastReadStore`, а HTTP API использует эти данные для ответа `/last-read`.

### DTO для HTTP API

DTO-слой отделяет внутренние структуры от HTTP-ответов.

Основные DTO:

- `RootResponseDto` - информация о приложении и ссылках;
- `HealthResponseDto` - состояние сервиса;
- `LastReadResponseDto` - последний результат чтения;
- `DataReadErrorDto` - ошибка при отсутствии данных;
- `EndpointLinksDto` - ссылки на основные эндпоинты.

### Контур БД

В проекте уже есть подготовка к хранению истории чтений:

- `PpuDbContext`;
- `RawReadHistoryEntry`;
- регистрация SQLite через EF Core.

Текущее состояние: контекст БД и сущность истории существуют, но основной поток опроса пока не записывает результаты в базу. Сейчас фактически используется только in-memory хранение последнего результата.

    |

## Основные потоки данных

### Поток опроса PLC

```text
appsettings.json
    -> PlcReaderOptions
    -> PollingWorker
    -> PlcReaderService
    -> PLC по Modbus TCP
    -> RawReadResult
    -> LastReadStore
```

Смысл потока: приложение берёт настройки подключения, периодически читает PLC и сохраняет последний результат в память.

### Поток HTTP-чтения последнего результата

```text
HTTP GET /last-read
    -> Minimal API
    -> LastReadStore
    -> LastReadResponseDto
    -> HTTP response
```

Если чтения ещё не было, API возвращает ошибку `No read yet.`.

### Поток диагностики

```text
HTTP GET /health
    -> Minimal API
    -> HealthResponseDto
    -> HTTP response
```

Этот поток не проверяет PLC и БД. Он показывает, что само HTTP-приложение запущено.

### Потенциальный поток истории

```text
RawReadResult
    -> RawReadHistoryEntry
    -> PpuDbContext
    -> SQLite
```

Этот поток логически подготовлен структурой проекта, но в текущей реализации ещё не подключён к `PollingWorker`.

## Высокоуровневые структуры данных

### `PlcReaderOptions`

Структура конфигурации чтения PLC. Используется при запуске приложения для передачи настроек в `PollingWorker` и `PlcReaderService`.

Через неё система понимает, к какому PLC подключаться, какой диапазон регистров читать и с каким интервалом выполнять опрос.

### `RawReadResult`

Основная рабочая структура результата чтения. Создаётся после каждого обращения к PLC.

Используется как промежуточный контракт между сервисом чтения, фоновым worker-ом, in-memory хранилищем и HTTP API. В ней передаётся общий итог операции: успешно ли выполнено чтение, какие данные получены и была ли ошибка.

### `RawReadHistoryEntry`

Структура для будущего хранения истории чтений в БД. Соответствует записи в таблице истории и должна использоваться, когда результат чтения нужно сохранить дольше, чем на время жизни процесса.

В текущей реализации сущность и EF Core-контекст уже есть, но запись истории из `PollingWorker` ещё не подключена.

### DTO HTTP API

DTO используются только на границе HTTP API. Они формируют ответы внешним клиентам и не должны становиться основной внутренней моделью системы.

Основные сценарии использования:

- `RootResponseDto` - ответ главной страницы API;
- `HealthResponseDto` - ответ health-check;
- `LastReadResponseDto` - внешний вид последнего результата чтения;
- `DataReadErrorDto` - сообщение об ошибке для клиента;
- `EndpointLinksDto` - список ссылок на основные эндпоинты.

## Текущие границы системы

Внутри системы:

- HTTP API;
- фоновый polling worker;
- сервис чтения PLC;
- in-memory состояние последнего чтения;
- доменные модели и DTO;
- подготовленный EF Core/SQLite слой.

Снаружи системы:

- PLC, доступный по Modbus TCP;
- HTTP-клиенты, которые читают состояние сервиса;
- SQLite-файл, если будет подключено сохранение истории.

## Направления дальнейшего развития

На основании текущей структуры наиболее естественные направления развития:

- подключить запись `RawReadResult` в SQLite через `RawReadHistoryEntry`;
- добавить API для чтения истории;
- добавить явную проверку PLC/БД в health-check;
- расширить декодирование регистров в прикладные сигналы;
- поддержать несколько PLC или несколько наборов регистров;
- выделить отдельный слой для маппинга между доменными моделями, DTO и сущностями БД.

## Архитектурный принцип для развития

Текущая структура лучше всего развивается вокруг простого конвейера:

```text
Configuration
    -> Polling
    -> Raw read
    -> Normalization
    -> Storage
    -> API
```

Новые функции желательно добавлять так, чтобы они не смешивали эти роли. Например, чтение PLC не должно знать о HTTP, API не должно напрямую работать с Modbus-клиентом, а хранение истории должно быть отдельным шагом после получения результата чтения.
